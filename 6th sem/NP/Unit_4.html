Unit 4: HTTP

HTTP (Hypertext Transfer Protocol)

What is HTTP?

HTTP is the foundation of data communication for the World Wide Web. It is a stateless, application-layer protocol for transmitting hypermedia documents, such as HTML. It was designed for communication between web browsers and web servers, but it can also be used for other purposes.

HTTP Methods

HTTP defines a set of request methods to indicate the desired action to be performed for a given resource. The most common methods are:

  • GET: Requests a representation of the specified resource. Requests using GET should only retrieve data.
  • POST: Submits an entity to the specified resource, often causing a change in state or side effects on the server.
  • PUT: Replaces all current representations of the target resource with the request payload.
  • DELETE: Deletes the specified resource.
  • HEAD: Asks for a response identical to that of a GET request, but without the response body.
  • OPTIONS: Describes the communication options for the target resource.
  • PATCH: Applies partial modifications to a resource.

Key Points to Remember: HTTP

  • The protocol of the web.
  • Stateless: each request is independent.
  • Uses request methods like GET, POST, PUT, DELETE.

Cookies

Define Cookies.

A cookie is a small piece of data that a server sends to a user's web browser. The browser may store the cookie and send it back to the same server with later requests. Cookies are used to maintain state in the otherwise stateless HTTP protocol. Common uses for cookies include session management (like keeping a user logged in), personalization, and tracking user behavior.

Java Program to Retrieve Cookie Information

This program demonstrates how to set up a CookieManager to automatically handle cookies and retrieve them after making a connection to a website.


import java.net.CookieHandler;
import java.net.CookieManager;
import java.net.CookiePolicy;
import java.net.HttpCookie;
import java.net.URL;
import java.net.URLConnection;
import java.util.List;

public class CookieRetriever {

    public static void main(String[] args) throws Exception {
        // Create a CookieManager to handle cookies
        CookieManager cookieManager = new CookieManager();
        cookieManager.setCookiePolicy(CookiePolicy.ACCEPT_ALL);
        CookieHandler.setDefault(cookieManager);

        // Connect to a website that sets cookies
        URL url = new URL("https://www.google.com");
        URLConnection connection = url.openConnection();
        connection.getContent(); // This triggers the interaction and cookie exchange

        // Retrieve the cookies from the cookie store
        List cookies = cookieManager.getCookieStore().getCookies();

        System.out.println("Cookies retrieved from " + url + ":");
        if (cookies.isEmpty()) {
            System.out.println("No cookies found.");
        } else {
            for (HttpCookie cookie : cookies) {
                System.out.println("--------------------------------");
                System.out.println("Name: " + cookie.getName());
                System.out.println("Value: " + cookie.getValue());
                System.out.println("Domain: " + cookie.getDomain());
                System.out.println("Path: " + cookie.getPath());
                System.out.println("Version: " + cookie.getVersion());
                System.out.println("Expires: " + (cookie.getMaxAge() != -1 ? cookie.getMaxAge() + " seconds" : "Session cookie"));
            }
        }
    }
}

Key Points to Remember: Cookies

  • Small pieces of data from a server, stored by the browser.
  • Used for session management, personalization, and tracking.
  • In Java, use CookieManager and CookieStore to handle them.

X-www-form-urlencoded Encoder and Decoder

Describe X-www-form-urlencoded Encoder and URL Decoder.

x-www-form-urlencoded is a format for encoding data in a URL's query string or in the body of an HTTP POST request. It is a way to send key-value pairs. The encoding scheme works as follows:

  • Alphanumeric characters are not changed.
  • Spaces are converted to plus signs (+).
  • Other non-alphanumeric characters are converted to a % symbol followed by the two-digit hexadecimal representation of the character's ASCII value.

The URLEncoder and URLDecoder classes in Java are used to perform this encoding and decoding.

Java Program for URL Encoding and Decoding


import java.net.URLDecoder;
import java.net.URLEncoder;
import java.nio.charset.StandardCharsets;

public class FormUrlEncoderDecoder {

    public static void main(String[] args) {
        try {
            String originalString = "This is a test string with spaces & special characters.";

            // Encode the string
            String encodedString = URLEncoder.encode(originalString, StandardCharsets.UTF_8.name());
            System.out.println("Original: " + originalString);
            System.out.println("Encoded:  " + encodedString);

            // Decode the string
            String decodedString = URLDecoder.decode(encodedString, StandardCharsets.UTF_8.name());
            System.out.println("Decoded:  " + decodedString);

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

Key Points to Remember: URL Encoding/Decoding

  • Used to safely transmit data in URLs.
  • Spaces become +, special characters become %xx.
  • Use URLEncoder.encode() and URLDecoder.decode().

HTTP Keep-Alive

What is the significance of controlling HTTP Keep-Alive?

HTTP Keep-Alive, also known as a persistent connection, is a feature that allows a single TCP connection to remain open for multiple HTTP requests and responses. The significance of controlling this feature is primarily related to performance and resource management:

  • Performance Boost: By keeping the connection open, it avoids the overhead of establishing a new TCP connection for each request, which involves a three-way handshake. This significantly reduces latency, especially for websites with many small resources like images and scripts.
  • Resource Management: While beneficial, persistent connections consume server resources (memory and sockets). On a very busy server, having too many open, idle connections can lead to resource exhaustion. Therefore, it's important to configure the Keep-Alive timeout (how long an idle connection is kept open) and the maximum number of requests per connection to balance performance and resource usage.

In Java, you can influence Keep-Alive behavior by setting system properties like http.keepAlive, http.maxConnections, http.keepAlive.time.server, and http.keepAlive.time.proxy.

Key Points to Remember: HTTP Keep-Alive

  • Reuses a single TCP connection for multiple HTTP requests.
  • Improves performance by reducing connection setup overhead.
  • Needs careful configuration to avoid server resource exhaustion.

Custom Cookie Policy

Java Program for a Cookie Policy that Blocks .gov Cookies

This program demonstrates how to create a custom CookiePolicy to filter cookies based on their domain.


import java.net.CookieHandler;
import java.net.CookieManager;
import java.net.CookiePolicy;
import java.net.HttpCookie;
import java.net.URI;

public class CustomCookiePolicy {

    public static void main(String[] args) {
        // Create a custom cookie policy
        CookiePolicy blockGovCookies = (uri, cookie) -> {
            // Block cookies from .gov domains
            if (cookie.getDomain() != null && cookie.getDomain().toLowerCase().endsWith(".gov")) {
                System.out.println("Blocking cookie from: " + cookie.getDomain());
                return false; // Do not accept the cookie
            }
            // Allow all other cookies
            return true;
        };

        // Set the custom policy
        CookieManager cookieManager = new CookieManager();
        cookieManager.setCookiePolicy(blockGovCookies);
        CookieHandler.setDefault(cookieManager);

        System.out.println("Custom cookie policy is active. It will block cookies from .gov domains.");

        // Example usage (simulation)
        try {
            URI govUri = new URI("https://www.usa.gov");
            HttpCookie govCookie = new HttpCookie("test", "123");
            govCookie.setDomain(".gov");

            URI comUri = new URI("https://www.google.com");
            HttpCookie comCookie = new HttpCookie("pref", "abc");
            comCookie.setDomain(".google.com");

            // Simulate the cookie manager deciding whether to accept cookies
            System.out.println("\nChecking a .gov cookie...");
            if (blockGovCookies.shouldAccept(govUri, govCookie)) {
                System.out.println("usa.gov cookie accepted (This should not happen).");
            } else {
                System.out.println("usa.gov cookie was correctly blocked.");
            }

            System.out.println("\nChecking a .com cookie...");
            if (blockGovCookies.shouldAccept(comUri, comCookie)) {
                System.out.println("google.com cookie was correctly accepted.");
            } else {
                System.out.println("google.com cookie was blocked (This should not happen).");
            }

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

Key Points to Remember: Custom Cookie Policy

  • Implement the CookiePolicy interface.
  • The shouldAccept(uri, cookie) method is where you define your logic.
  • Return true to accept the cookie, false to reject it.
  • Set your custom policy on a CookieManager.
