Group B
Explain android architecture briefly with a suitable diagram. What is a surface manager library? How is DVM different from ART?

Android architecture is organized into five main layers:

  1. Linux Kernel: The base layer that manages hardware drivers (Camera, WiFi, Flash memory, Display, Bluetooth, etc.) and power management.
  2. Hardware Abstraction Layer (HAL): Provides standard interfaces that expose device hardware capabilities to the higher-level Java API framework.
  3. Android Runtime (ART/DVM) & Native C/C++ Libraries: Contains core libraries and the runtime environment.
    • Native Libraries: Includes WebKit (browser engine), SQLite (database), OpenGL (graphics), and Surface Manager.
  4. Java API Framework: Provides the building blocks for creating apps (Activity Manager, Resource Manager, Notification Manager, Content Providers, View System).
  5. System Apps: The top layer containing pre-installed apps like Email, SMS, Calendar, Maps, and Browser.
[Diagram Description: Stack from bottom to top: Linux Kernel -> HAL -> Native Libraries & Android Runtime -> Java API Framework -> System Apps]

Surface Manager: It is a library that manages access to the display subsystem and seamlessly composites 2D and 3D graphic layers from multiple apps onto a single screen.

DVM vs. ART:

Feature Dalvik Virtual Machine (DVM) Android Runtime (ART)
Compilation Just-In-Time (JIT) compilation (code compiled during execution). Ahead-Of-Time (AOT) compilation (code compiled during installation).
Battery Life Higher battery consumption due to frequent JIT compilation. Better battery life as compilation happens once.
App Launch Slower app launch. Faster app launch.
Storage Requires less storage space. Requires more storage space for compiled binaries.
Group B
Draw the hierarchical directory structure of an android application.

A typical Android project (in Android Studio) has the following structure:

MyApplication/
├── app/
│   ├── manifests/
│   │   └── AndroidManifest.xml  (App configuration, permissions)
│   ├── java/
│   │   └── com.example.app/
│   │       └── MainActivity.java (Source code)
│   └── res/ (Resources)
│       ├── drawable/ (Images/Icons)
│       ├── layout/ (XML layout files like activity_main.xml)
│       ├── mipmap/ (App icons for different densities)
│       └── values/ (strings.xml, colors.xml, styles.xml)
├── Gradle Scripts/
│   ├── build.gradle (Project level)
│   └── build.gradle (Module: app)
Group B
How to create AVD's?

An Android Virtual Device (AVD) is an emulator configuration that allows you to model an actual Android device. Steps to create one in Android Studio:

  1. Open Android Studio and go to Tools > Device Manager.
  2. Click Create Device.
  3. Select Hardware: Choose a device profile (e.g., Pixel 6, Nexus 5X) and click Next.
  4. Select System Image: Choose an Android version (API level) to install on the device (e.g., Tiramisu, UpsideDownCake). Download it if necessary. Click Next.
  5. Verify Configuration: Change the AVD name if desired, adjust orientation, and click Finish.
  6. The AVD will now appear in the Device Manager. Click the Play button to start the emulator.
Group B
What does an android APK file contain?

An APK (Android Package) file is a ZIP-formatted file that contains everything needed for an Android app to run. It includes:

  • META-INF/: Contains the jar signature and a list of resources.
  • lib/: Contains platform-specific compiled code (native libraries like .so files).
  • res/: Contains resources not compiled into resources.arsc (like images).
  • assets/: Contains application assets (raw files like fonts or data files).
  • AndroidManifest.xml: The binary version of the manifest file.
  • classes.dex: The compiled Java/Kotlin classes in Dalvik Executable (DEX) format.
  • resources.arsc: Contains compiled resources (like strings and layouts) used by the app.
